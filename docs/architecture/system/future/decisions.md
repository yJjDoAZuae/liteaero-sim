# Architectural Decisions and Open Questions — Future State

---

## Decisions Made

| Decision | Choice | Rationale |
| --- | --- | --- |
| LiteAero Sim scope | Simulation plant only | Separates flight-code reuse from simulation-specific code; enables flight software deployment without simulation dependencies |
| LiteAero Flight as separate component | Yes — not a LiteAero Sim subsystem | Flight code must meet flight-software standards; including it in a simulation library creates wrong-level dependencies |
| MAVLink as baseline external protocol | Yes, with custom extension where needed | ArduPilot and PX4 compatibility; QGroundControl integration; extensible |
| Autopilot / Navigation boundary | Navigation is outside Autopilot boundary | Separation of estimation and control; enables navigation filter replacement |
| Perception functions | Accommodated in architecture, not yet designed | Identifies kinds of perception capabilities; sets requirements for future integration without redesign |
| Docker containerization for SITL verification | LiteAero Sim and LiteAero Flight in separate containers for verification venues; direct-call permitted for development only | Ensures the flight code under test executes in the same runtime environment as the active flight software load; the container boundary provides hard enforcement of the ICD-8 architectural boundary with no shared memory or in-process calls possible |
| LiteAero Flight repository structure | Separate `liteaero-flight` repository; shared interface data types (AircraftCommand, NavigationState, sensor measurement structs) live in `liteaero-flight`; LiteAero Sim takes a versioned dependency on `liteaero-flight` | Repo boundary structurally enforces the one-way dependency rule (simulation may depend on flight code; flight code must not depend on simulation); shared types belong in `liteaero-flight` because flight code defines and owns them |
| C++ namespace structure | `liteaero` top-level organizational namespace; second-level namespaces: `liteaero::perception`, `liteaero::mission_autonomy`, `liteaero::nav`, `liteaero::guidance`, `liteaero::autopilot`, `liteaero::simulation`; cross-cutting infrastructure namespaces: `liteaero::log` (ILogger and logging sinks), `liteaero::control` (DynamicElement, SisoElement, Filter hierarchy, Integrator, Derivative, RateLimit, Limit, SISOPIDFF), `liteaero::terrain` (TerrainVertex, TerrainFacet, TerrainLod, TerrainTile, GeodeticPoint, GeodeticAABB, LocalAABB, V_Terrain) | Org-scoped top level avoids collision in any multi-library build; subsystem namespaces follow the architectural boundary structure; `liteaero::simulation` applies to LiteAero Sim code, the remaining namespaces to LiteAero Flight; three infrastructure namespaces are kept separate because functional subsystems compose them independently — autopilot needs control but not terrain, perception needs terrain but not control, and future recomposition patterns may require logging without either |
| CMake build target structure | One CMake library target per subsystem, matching the C++ namespace: `liteaero::nav`, `liteaero::guidance`, `liteaero::autopilot`, `liteaero::perception`, `liteaero::mission_autonomy`, `liteaero::simulation`; three infrastructure targets matching the infrastructure namespaces: `liteaero::log`, `liteaero::control`, `liteaero::terrain`; shared interface types (AircraftCommand, NavigationState, sensor measurement structs) require a separate target, name TBD (see OQ-12) | Enables deployment targets to link only the subsystems and infrastructure they need; fine-grained infrastructure targets avoid forcing transitive dependencies on configurations that do not require them; elements are expected to be recomposed in future use cases beyond those currently defined, making granular linking worthwhile; consistent with the C++ namespace structure |
| SITL boundary enforcement model | Two SITL environments: (1) development SITL — single process, direct function call, no boundary enforcement, for rapid iteration; (2) integration SITL — containerized (UC-2b), strict ICD-8 boundary enforcement, required gate before HITL | Eliminates the class of errors that appear at HITL but are not caught in single-process development SITL; transport, serialization, and timing errors that would only manifest at the container or hardware boundary are exposed at integration SITL instead |
| Control element infrastructure ownership | `DynamicElement`, `SisoElement`, `Filter` hierarchy, `Integrator`, `Derivative`, `RateLimit`, `Limit`, `SISOPIDFF`, and related control elements belong in `liteaero-flight` under `liteaero::control`, not `liteaero-sim` | These are general-purpose discrete-time control and DSP infrastructure with no dependency on any simulation concept; their current location in the LiteAero Sim repository is an artifact of development order; both simulation and flight code link them from `liteaero-flight` |
| Logging interface ownership and deployment model | `ILogger` interface and logging sinks belong in `liteaero-flight` under `liteaero::log`; flight code provides its own logger implementation (`FlightLogger`) for HITL and real-flight deployments; the simulation `Logger` implements `ILogger` and is injected into flight code components during SITL | Flight code components log independently of simulation in HITL and real-flight contexts; dependency inversion through `ILogger` makes flight code components agnostic to which logger is injected; `liteaero::log` is kept separate from `liteaero::control` so components that log but are not `DynamicElement` subclasses do not take a transitive dependency on the filter and control hierarchy |
| Terrain type split | Common terrain mesh elements (`TerrainVertex`, `TerrainFacet`, `FacetColor`, `TerrainLod`, `TerrainTile`, `GeodeticPoint`, `GeodeticAABB`, `LocalAABB`) and the `V_Terrain` query interface belong in `liteaero-flight` under `liteaero::terrain`; simulation-specific terrain machinery (`TerrainMesh`, `LodSelector`, `FlatTerrain`, `TerrainCell`, `MeshQualityVerifier`, terrain file I/O) stays in `liteaero::simulation` | Perception, relative navigation, and guidance depend on terrain data at runtime; they program against `V_Terrain` and the mesh element types without knowing the simulation terrain implementation |
| Namespace adoption timing | Single-step migration deferred to the repo split milestone; all namespace changes (infrastructure and shared types to their decided `liteaero-flight` namespaces; simulation code to `liteaero::simulation`) are executed as one coordinated event | Avoids a prolonged half-migrated codebase; consistent with the no-backward-compat policy; the repo split is a near-term planned milestone; all target namespaces are now final |
| Module communication interfaces use plain value structs | All data types that cross subsystem or ICD boundaries are lightweight plain value type structs with no methods beyond accessors; no polymorphism, no lifecycle, no serialization machinery in shared interface types | Keeps interfaces deployable on constrained flight hardware; avoids hidden dependencies across the ICD-8 boundary; consistent with the snapshot pattern required for container-network and network transport |
| 6DOF aerodynamic coefficient convention and model interface | Body-axis stability-derivative convention (CX, CY, CZ, Cl, Cm, Cn) is the baseline; architectural flexibility for wind-axis coefficients (CL, CD, CY, Cl, Cm, Cn) is preserved by placing the coefficient model behind an abstract `V_AeroModel` interface; `Aircraft6DOF` depends only on `V_AeroModel`, which produces forces and moments in body frame; body-to-wind or wind-to-body transformations are the responsibility of the concrete implementation, keeping the 6DOF integrator agnostic to the coefficient axis convention | The 6DOF integrator requires forces and moments in body frame regardless of the source convention; hiding the convention behind an interface costs nothing at runtime and avoids a future redesign if wind-axis coefficients are required |
| Perception functions to accommodate | Three categories: (1) vision-based navigation — synthetic image generation from terrain and scene model, processed by `liteaero::perception` for position/attitude estimation; (2) laser AGL sensors and lidar — synthetic ranging against terrain model for terrain clearance and 3D point cloud; (3) line-of-sight analysis for communication links — terrain-occluded RF link quality assessment; all three require the terrain model at runtime and use `V_Terrain` as the underlying geometry source | Sets requirements on simulation sensor models and the `liteaero::perception` subsystem design; all three categories are proposed and not yet designed |
| Game engine for real-time visualization | Godot 4 (MIT license, open source); transport is UDP to a fixed local port at simulation rate (typically 100–400 Hz); GLB files (glTF 2.0) used for terrain asset delivery; live simulation data injected via GDExtension plugin; Godot 4 and glTF 2.0 share the same axis convention (X=East, Y=Up, Z=-North), so GLB terrain files import without root-node rotation | Decided during terrain mesh implementation (Step 11/12); documented in `docs/architecture/terrain.md §Game Engine Integration` |
| Aircraft model output interface | All aircraft model implementations (`Aircraft`, `Aircraft6DOF`, and any future variants) produce `KinematicStateSnapshot` as their output; flight code, sensors, and loggers program against this common output type and are agnostic to which dynamics model is underneath | Enables side-by-side comparison between models and transparent substitution in any simulation topology |
| 6DOF aircraft model command interface and control topology | `Aircraft6DOF` accepts `SurfaceDeflectionCommand` (control surface deflection angles + per-motor throttle), not the load-factor `AircraftCommand`; a `FBWController` element bridges load-factor commands to surface deflections and pairs with `Aircraft6DOF` as a drop-in replacement for the existing `Aircraft` for side-by-side comparison; ArduPilot and PX4 simulations connect directly to `Aircraft6DOF` via `SurfaceDeflectionCommand` without the FBW bridge, because their inner-loop control topology differs fundamentally from the load-factor FBW interface enforced by the existing `Aircraft` | Inner-loop control topology for ArduPilot/PX4 (attitude/rate → surface deflections) is topologically incompatible with the existing Aircraft FBW interface (load-factor commands → allocator → α, β); the 6DOF model must accept both topologies transparently; the FBWController bridge preserves the existing interface for LiteAero Sim autopilot development and comparison testing |
| Simulation and flight code envelope models are independent | The simulation models ultimate limits at which (1) emulated FBW imposes limits, (2) the aircraft departs controlled flight, (3) a structural failure occurs, or (4) an ultimate performance limit (thrust, etc.) is reached; the flight code (autonomy, guidance, autopilot) defines its own more stringent operational envelope limits in mathematical forms appropriate to those functions, which are not yet defined; `AirframePerformance`, `LiftCurveModel`, `LiftCurveParams`, `AeroPerformanceConfig`, and `LoadFactorAllocator` are simulation-internal types and are not shared with flight code | The two envelope formulations serve different purposes and will not be mathematically equivalent; sharing simulation envelope types with flight code would create a false coupling between simulation fidelity choices and operational limit definitions |
| HITL command path latency and simulation timestep rate | Maximum design ceiling: 10 ms command path latency, 100 Hz simulation timestep rate; many deployment configurations will operate at significantly lower performance (lower rate, higher latency tolerance); the architecture does not mandate 100 Hz — `SimRunner` timestep rate is a deployment configuration parameter; 10 ms latency bound is a ceiling that the ICD-8 network transport must not exceed in high-rate configurations | Sets an upper bound that the network transport and hardware must meet for demanding HITL use cases; lower-performance configurations (longer timestep, relaxed latency) are explicitly accommodated and do not require architectural changes |
| `LandingGear` integration model and fidelity target | Single `LandingGear` design, internal to both `Aircraft` and `Aircraft6DOF`; produces `ContactForces` (plain value struct: `force_body_n` + `moment_body_nm` as `Eigen::Vector3f` each, plus `weight_on_wheels` boolean flag); `LandingGear::step()` accepts `KinematicStateSnapshot`, `V_Terrain` (ground elevation query), `nose_wheel_angle_rad`, `brake_left_nd`, and `brake_right_nd`; subsystem implements 2nd-order suspension per strut (spring-damper for bounce modeling), tyre contact forces (vertical, longitudinal, lateral), and WOW status; **`Aircraft` (load-factor model)**: passes `force_body_n[Z]` and `force_body_n[Y]` to `LoadFactorAllocator` as disturbance terms alongside aerodynamic and propulsion forces; applies `moment_body_nm` (including ground steering yaw) directly to the kinematic update, bypassing the allocator; **`Aircraft6DOF`**: applies all six components of `ContactForces` directly to the equations of motion; ground steering (nose wheel) and differential braking produce moments outside FBW authority in both models and bypass the allocator because they are separate actuator systems; **design target use case**: developmental verification of autotakeoff and autolanding functions — ground contact, bounce, WOW establishment, rollout heading control, taxi; the design does not target: suspension parameter sizing, precise aircraft-on-ground detection algorithm design, or control surface mixing design; **validity bound for `Aircraft` ground operations**: FBW authority exceedance from gear-induced pitch and roll moments is not modeled; high-speed ground dynamics scenarios that depend on authority limit simulation (e.g., crosswind landing rollout at or above rotation speed) require `Aircraft6DOF` | Single design maximizes commonality; for the autotakeoff/autoland verification use case, the load-factor model's emulation of FBW behavior is sufficiently consistent because that use case verifies guidance and autopilot sequencing algorithms, not FBW authority sizing; ground steering and differential braking bypass the allocator because they are separate actuator systems outside FBW authority; `Aircraft6DOF` provides full fidelity for scenarios requiring pitch and roll authority limit simulation |

## Open Questions Requiring Resolution

All remaining open questions are design-phase questions. They are not resolvable at the
architecture definition stage; they are tracked here and will be resolved at the appropriate
point in the software design phase for each affected component.

### Design Phase
